# Развертывание стабилизированной версии

Изменения совместимы с текущим интерфейсом и существующими Avito/iCal-ссылками. Для включения постоянного и атомарного режима нужно применить SQL-миграцию до публикации новой версии сервера.

## Обязательный порядок

1. Создать резервную копию Supabase Database и убедиться, что Storage bucket с фотографиями доступен.
2. В Supabase SQL Editor последовательно применить отсутствующие миграции из `src/sql` вплоть до `005_external_calendars.sql`.
3. Одним запуском применить `src/sql/006_stability_hardening.sql`.
4. Развернуть приложение на VPS/Coolify.
5. Проверить `GET /health` — ожидается HTTP 200.
6. Проверить `GET /ready` — ожидается HTTP 200 и `migration_006: true`.
7. Войти в админку заново, чтобы получить новую сессию и CSRF-cookie.
8. Открыть каждый домик и убедиться, что существующие iCal/Avito-ссылки отображаются. При первом чтении старые JSON-ссылки автоматически импортируются в Supabase.
9. Нажать «Синхронизировать» у одного источника и проверить занятые даты на публичном календаре.
10. Создать тестовую заявку на свободные даты, сравнить сумму в публичной форме, базе и админке, затем отменить заявку.

## Проверка миграции в SQL Editor

```sql
select key, updated_at from app_config order by key;

select id, cabin_id, source_name, is_active, last_sync_status
from external_calendar_sources
order by cabin_id, created_at;

select routine_name
from information_schema.routines
where routine_schema = 'public'
  and routine_name in (
    'create_booking_atomic',
    'replace_external_bookings',
    'save_external_calendar_sources',
    'save_cabin_full'
  );
```

После первого открытия административных разделов в `app_config` должны появиться ключи `settings`, `amenities`, `mainpage`, `tags`, `cabin_tags`, `extra_services`, `house_items` и `guest_notes` по мере обращения к ним.

## Важные эксплуатационные правила

- Не использовать локальные файлы `src/data/*.json` как резервную копию актуальных production-данных. После миграции источником истины является Supabase.
- Не удалять вручную файлы в Supabase Storage. Админка удаляет отвязанные фотографии сама.
- Перед каждым деплоем запускать `npm test` и проверять `/ready` после деплоя.
- Для диагностического запуска без фоновой синхронизации задавать `DISABLE_BACKGROUND_JOBS=true`.
- При ошибке `/ready` не принимать новые бронирования до восстановления соединения с Supabase или применения миграции.

## Что сохраняется без изменений

- текущий публичный сценарий выбора домика и дат;
- отображение индивидуальных цен по датам;
- существующие Avito/iCal URL, включая `webcal://`;
- правило: выбранная дополнительная услуга прибавляется к заявке один раз;
- внутренний триггер защиты от пересекающихся бронирований;
- Telegram-уведомления и чат.
